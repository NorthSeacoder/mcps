# Feature Specification: Hermes DB MCP Platform Revival

**Workspace**: `mcps`  
**Created**: 2026-05-27  
**Status**: Clarified  
**Input**: 用户希望复活旧的 `mcps` 仓库，将 `hermes-db` 迁入统一管理，并打通原有部署流程；同时要求 NAS 私有配置通过 `.gitignore` 控制，后续所有 MCP 都沿用同一套部署规范，差异只体现在构建镜像和 NAS 拉取镜像名。

## Background

当前仓库 `mcps` 仍保留旧的 pnpm workspace + turbo 结构，但内容只剩模板式 MCP 项目，缺少面向实际生产的服务组织方式、部署约定和 NAS 私有配置隔离。另一方面，`hermes-db-mcp` 已是独立 Python MCP 服务，具备领域工具、Docker Compose 和 SSE 部署能力，但它和 `mcps` 的工程规范没有统一。

本 feature 的目标不是把 NAS 上既有运行态一次性重切，而是先把 `mcps` 复活为统一的 MCP 源码与部署仓，再逐步接管 `hermes-db` 的源码和发布链路。NAS 上的私有信息、镜像地址、环境变量和反代细节必须保持本地化，不进入开源仓。

## Goals

- 将 `mcps` 恢复为可持续维护的 MCP 平台仓。
- 把 `hermes-db` 迁入 `mcps` 统一管理，但先只覆盖源码、构建、发布和部署脚本，不强制立即切换现有 NAS 运行态。
- 统一所有后续 MCP 的部署流程约定。
- 通过 `.gitignore` 将 NAS 私有配置与本地环境覆盖文件隔离。
- 为后续在 `mcps` 仓内启动 agent、管理多个 MCP 服务和编排部署流程预留清晰边界。

## User Scenarios & Testing

### User Story 1 - 复活 mcps 平台仓 (Priority: P1)

作为仓库维护者，我希望 `mcps` 不再只是旧模板仓，而是可以承载真实 MCP 服务、共享脚本和部署规范的平台仓。

**Why this priority**: 平台仓是所有后续工作的基础，不复活则其他 Story 无法推进。

**Acceptance Scenarios**:

1. 根目录保留 pnpm workspace / turbo 的统一管理方式。
2. 仓库内有明确的服务目录约定，可容纳多个 MCP 服务。
3. 新 MCP 的创建、构建、开发、部署命令沿用同一套约定。

**Edge Cases**:

- **US1-E1** 旧模板仓脚本（如 `create-server.js`）与新平台约定冲突时，应以新约定为准并更新脚本，不保留不兼容的旧行为。
- **US1-E2** 根工作区命令在某个服务缺少依赖时，应只影响该服务的构建，不阻塞其他服务。

### User Story 2 - 迁入 hermes-db (Priority: P1)

作为部署者，我希望 `hermes-db` 的源码、Docker 构建和部署流程纳入 `mcps`，以便和未来其他 MCP 复用相同的工程规范。

**Why this priority**: `hermes-db` 是首个真实迁入的服务，验证平台仓约定可行性，必须与 P1 平台仓基础同步落地。

**Acceptance Scenarios**:

1. `hermes-db` 的源码和部署描述可以在 `mcps` 中找到。
2. 构建镜像、推送镜像、NAS 拉取镜像的流程被统一定义。
3. 现有 NAS 运行态可以先保持不变，迁移通过明确切换步骤完成。

**Edge Cases**:

- **US2-E1** 迁移过程中如果发现 `hermes-db` 原有 Dockerfile / compose 与平台公共约定冲突，应优先保留原有可运行配置，并在文档中明确差异点，不强行改造。
- **US2-E2** 如果迁移后构建产生的镜像与 NAS 现行容器镜像不一致，必须保证两者可并存，旧运行态不被覆盖。
- **US2-E3** 源码迁入过程中如果发现 `hermes-db` 依赖某些只在原仓库下可用的脚本或环境变量，应在迁入时显式记录到服务 README，避免隐式断链。

### User Story 3 - NAS 私有配置隔离 (Priority: P1)

作为仓库维护者，我希望 NAS 上的密钥、地址、镜像引用和本地覆盖配置不进入开源仓，以免泄露环境差异。

**Why this priority**: 私有信息一旦提交即视为泄露，必须在首次提交平台仓改造之前就建立隔离规则，是 P1 的安全底线。

**Acceptance Scenarios**:

1. NAS 专用配置文件通过 `.gitignore` 排除。
2. 仓库内提供私有配置模板或示例文件。
3. 开源仓中的默认配置不会依赖 NAS 私有值才能通过本地开发。

**Edge Cases**:

- **US3-E1** 如果开发者新增了一类未在 `.gitignore` 中覆盖的本地覆盖文件，提交前应能通过约定（命名规范 + 文档说明）被识别和补入忽略规则。
- **US3-E2** 模板文件中如果出现示例值与真实 NAS 值高度相似的情况，必须使用明显占位符（如 `your-nas-host`）以避免误用。
- **US3-E3** 如果 git 历史中已有早期意外提交的私有信息，应在迁移说明中标注此风险并指引清理路径，但本次范围只防止新增泄露。

### User Story 4 - 统一未来 MCP 部署 (Priority: P2)

作为仓库维护者，我希望后续新增 MCP 时只需要替换镜像构建目标和 NAS 拉取镜像名，而不用重新设计一套部署流程。

**Why this priority**: 这一项收益体现在后续每个新 MCP 接入时，本期只需建立骨架并验证一次，不阻塞首期 `hermes-db` 迁入，故定为 P2。

**Acceptance Scenarios**:

1. 新 MCP 能复用同一套构建与部署骨架。
2. 不同服务之间只在镜像名、构建上下文、运行参数上存在差异。
3. 部署文档能区分公共流程和服务专属参数。

**Edge Cases**:

- **US4-E1** 当某个新 MCP 的运行参数明显超出公共骨架表达能力（如需要 GPU、特殊网络、长任务）时，骨架应允许服务级覆盖文件扩展，而不是强行塞进公共流程。
- **US4-E2** 如果两个服务对同一约定（如端口、卷路径）有冲突默认值，部署文档必须能让冲突显式可见，避免静默覆盖。

## Requirements

### Functional Requirements

- **FR-001**: `mcps` 必须恢复为可扩展的 MCP 平台仓，而不是只保留旧模板示例。
- **FR-002**: `mcps` 必须包含统一的服务组织结构，支持多个 MCP 服务共存。
- **FR-003**: `hermes-db` 必须迁入 `mcps` 的统一管理边界，至少覆盖源码、构建、部署脚本和文档。
- **FR-004**: 迁移首期必须保留现有 NAS 运行态可用，不强制在同一阶段完成容器/数据卷/域名切换。
- **FR-005**: 必须定义一套统一的 MCP 部署流程，后续所有 MCP 复用该流程，只替换服务专属镜像与运行参数。
- **FR-006**: NAS 私有配置必须通过 `.gitignore` 和本地覆盖文件隔离，不进入开源提交。
- **FR-007**: 必须保留本地开发与 NAS 部署的双路径约定。
- **FR-008**: 必须为后续在 `mcps` 仓内启动 agent 预留文档和目录边界。

### Non-Functional Requirements

- **NFR-001**: 迁移过程应以可回滚为前提，避免一次性切断现有运行态。
- **NFR-002**: 公共仓中不得包含 NAS 密钥、内网地址、私有镜像仓路径等敏感值。
- **NFR-003**: 新增 MCP 的工程接入成本应明显低于当前独立维护多个散仓的方式。
- **NFR-004**: 部署文档必须能区分”公共约定”和”NAS 私有覆盖”。

### Quality Attributes

| 属性 | 目标 | 为什么重要 | 验收 / 证据 | 是否阻塞 plan |
|------|------|------------|-------------|----------------|
| 可演进性 | 后续新增 MCP 时只需替换服务目录和镜像名，不改平台核心 | 仓库会持续接入更多 MCP，平台层若需反复改造将拖慢交付 | 在 plan 中能写出”新增最小服务时不改平台核心脚本”的接入路径 | 否 |
| 安全性 | NAS 私有信息不进入开源仓的任何提交 | 仓库可能开源或共享，私有泄露不可逆 | `.gitignore` 规则 + 模板文件审查 + 提交前检查约定 | 是 |
| 可回滚性 | 首期迁移不破坏既有 NAS 运行态，源码迁移与运行态切换解耦 | 现有 NAS 运行态承载实际业务，强切风险高 | 迁移前后能分别构建与部署，旧运行态保持可用 | 是 |
| 可维护性 | 平台仓职责分层清晰（platform / service / overlay） | 多人协作和后续扩展都依赖结构清晰 | `packages/` `deploy/` `docs/` 各自有独立 README 说明职责 | 否 |
| 一致性 | 不同 MCP 的部署流程在公共部分保持一致，差异显式表达 | 否则碎片化会重新出现，平台仓失去意义 | 部署文档明确区分公共流程与服务专属参数 | 否 |

### Key Entities

- **MCP Service**: 单个可独立构建与部署的 MCP 服务。
- **Platform Repo**: 统一管理多个 MCP 的源码仓与部署规范。
- **NAS Profile**: 仅存在于本地环境的私有部署配置。
- **Image Contract**: 某个服务对应的镜像名、构建目标和拉取约定。

## Out of Scope

- 不在首期强制迁移现有 NAS 运行中的容器、数据卷和反代配置。
- 不在首期修改 `hermes-db-mcp` 的业务工具契约。
- 不在首期把 `content-orchestrator-agent` 一并纳入 `mcps`。
- 不在首期统一所有可能存在的非 MCP 项目。
- 不在首期实现新 MCP 的自动化生成器重写，先保留最小可用模板即可。

## Success Metrics

- `mcps` 能作为真实项目继续维护，而不是停留在旧模板状态。
- `hermes-db` 的源码和部署规范可以在 `mcps` 中被定位和管理。
- NAS 私有信息不出现在开源仓库提交历史中。
- 新增 MCP 时能复用现有仓库结构和部署流程，而不是重新起一套工程。

## Clarified Decisions

- **D1**: 本 feature 只迁源码与部署流程，不把现有 NAS 运行态强制一次性切换。
- **D2**: 所有 NAS 私有信息走 `.gitignore` 管控的本地覆盖文件。
- **D3**: 文档、spec、plan 和后续 agent 入口都放在 `mcps` 项目下。
- **D4**: 后续所有 MCP 共享同一套部署骨架，差异只在构建镜像和 NAS 拉取镜像名。
